# Where things live

How the project is organized across the three places team work happens:
**the repo**, **the shared Drive**, and **Kaggle**. Read this first when
you can't find something or aren't sure where to put something.

## TL;DR — three places

| Where | What it holds | Format |
|---|---|---|
| **Repo** (this GitHub repo) | code, configs, team-facing docs | text, in git |
| **Drive** (private — shared with the team) | raw / binary / large / external artifacts | PDFs, slides, datasets, screenshots |
| **Kaggle** ([Track A](https://www.kaggle.com/competitions/ml-gen-x-bioreasoning-challenge-track-a) · [Track B](https://www.kaggle.com/competitions/ml-gen-x-bioreasoning-challenge-track-b)) | challenge data, leaderboard, official submission | platform-managed |

Rule of thumb when you don't know where something goes:

- Is it **code or a concise text doc** that should diff cleanly? → **Repo**
- Is it **raw, binary, large, or external**? → **Drive**
- Is it **competition data or a submission**? → **Kaggle**

## Repo

GitHub: https://github.com/jmsung/bio-reasoning

```
.
├── src/bio_reasoning/    importable library code (data, features, models, training, eval, trial_loop, utils)
├── scripts/              entry points (prepare_data.py, train.py, predict.py, make_submission.py)
├── configs/              one YAML per experiment
├── notebooks/            exploration (numbered: 01-eda.ipynb, …)
├── tests/                pytest
├── docs/                 team-authored project docs (this file, challenge.md, roadmap, conventions)
├── knowledge/            ingested + distilled external knowledge — managed by `/wiki-*` skills
├── data/                 gitignored except READMEs — local working copies only
├── models/               gitignored — local checkpoints
├── outputs/              gitignored — local run artifacts
└── submissions/          gitignored — local submission files
```

Rule: if it would be `import`ed from a notebook or another script, it lives
in `src/`. Otherwise `scripts/`. Never duplicate functions across scripts.

Canonical docs in `docs/`:
- `challenge.md` — what the challenge is, tracks, entry decision
- `where-things-live.md` — this file

## Drive (shared with team)

Root: private — shared with the team

```
bio-reasoning-2026/
├── 00-logistics/    meeting notes, team agreements, deadlines, contact info
├── 01-challenge/    official PDFs, rules, organizer comms, Kaggle screenshots
├── 02-papers/       literature PDFs we want everyone to read
├── 03-data/         raw dumps / external refs too big or licensed-not-to-commit
├── 04-experiments/  run reports, eval tables, figures we want to share informally
├── 05-writeups/     drafts of workshop submission, slides, posters
└── 06-scratch/      unsorted — promote out, don't let it grow
```

Drive is the **raw + binary** layer. If a teammate would want to read it on
their phone or print it, it probably belongs here.

## Kaggle (competition platform)

- **Data** — pulled with the Kaggle CLI; do not commit to the repo. Local
  copies live in `data/raw/` (gitignored).
- **Submissions** — generated by `scripts/make_submission.py`, written to
  `submissions/` (gitignored), uploaded via Kaggle CLI.
- **Leaderboard** — source of truth for scoring. We do not mirror it.

## Heuristics — common situations

| You have | Goes in |
|---|---|
| A paper PDF you want the team to read | Drive `02-papers/` |
| A `train.py` flag you added | repo `scripts/train.py` + update `README.md` |
| A team meeting summary | Drive `00-logistics/` |
| A Kaggle submission CSV | local `submissions/`, upload to Kaggle |
| A plot for a slide deck | Drive `05-writeups/` |
| A design note / decision record | repo `docs/` |
| A decision about which track we enter | repo `docs/challenge.md` |

## Keeping this honest

If you move something or add a new layer, update this file in the same PR.
Stale orientation docs are worse than missing ones.
